Summary #

Add layered supply-chain defenses to both the published catalog (printing-press-library) and its upstream generator (cli-printing-press), targeting the attack shapes observed in four real-world May 2026 incidents: Axios, TanStack mini-Shai-Hulud, node-ipc, and BufferZoneCorp.

The defense combines two layers, applied identically across both repos:

  • Greptile rules (judgment): six new entries in each repo's greptile.json covering GitHub Actions trust patterns, Go-module supply-chain shapes, npm-wrapper lifecycle scripts, Go-module-path drift on existing CLIs, and Go-source credential reads.
  • Deterministic Python scan (mechanical hard-fail): a small script wired into a new verify-supply-chain.yml workflow that blocks PRs introducing the highest-confidence attack signals. Patterned after the existing .github/scripts/verify-publish-package/ shape.

The plan lands in two coordinated PRs (one per repo) by an author with admin on both, with the published-library PR merging first so false-positive calibration informs the generator-repo mirror.

Problem Frame #

The printing-press-library repo publishes a catalog of Go CLIs (119 today) plus an npm installer wrapper (@mvanhorn/printing-press), and is fed by the cli-printing-press generator. The existing PR-time gates (verify-library-conventions.yml, verify-skills.yml, govulncheck.yml, Greptile review) cover canonical CLI shape and known-CVE Go dependencies. They do not cover supply-chain attack shapes that have hit comparable ecosystems in the last two months.

Current state audit (origin/main, both repos)

Performed on 2026-05-17 against fresh origin/main fetches. Findings:

  • SAFE printing-press-library has exactly one pull_request_target workflow (.github/workflows/greptile-policy-gate.yml). Permissions are checks: read, contents: read, issues: write, pull-requests: write; it does not check out PR head code and does not grant id-token. It only makes API calls.
  • SAFE printing-press-library grants id-token: write only in .github/workflows/npm-publish.yml, the OIDC Trusted Publishing job for the npm wrapper. This is the recommended posture.
  • SAFE cli-printing-press has exactly one pull_request_target workflow (.github/workflows/conversation-resolution-check.yml). Permissions are contents: read, pull-requests: read; no checkout of PR head, no id-token.
  • SAFE cli-printing-press has no id-token: write anywhere; release.yml uses keyed cosign rather than keyless OIDC signing.
  • EXISTS library/food-and-dining/ordertogo/go.mod has three replace directives, all pointing at local relative paths (./third_party/...). These pre-date this plan and are legitimate vendoring; the rule design accounts for them (see KD5).
  • CLEAN No GOPROXY, GOFLAGS, or GONOSUMCHECK overrides in any workflow in either repo.
  • CLEAN Every library/<cat>/<slug>/go.mod spot-checked uses the canonical module github.com/mvanhorn/printing-press-library/library/<cat>/<slug> form.

Audit conclusion: both repos are in good baseline posture. The new rules are forward-looking gates, not retroactive fixes. No existing code needs to change for the audit to clear — only new entries on PRs will trip the gates.

Implicit assumptions documented

  • The module_path attack surface lives in library/<cat>/<slug>/go.mod's module directive (verified — there is no module_path field in .printing-press.json).
  • The npm wrapper's blast radius dominates per-CLI Go binary blast radius: a compromised @mvanhorn/printing-press reaches every user via npx on install, whereas a single compromised Go CLI only reaches users who installed that one slug.
  • The generator repo's bugs that produce broken published CLIs are not in scope here; this plan defends against malicious PRs, not template regressions.

Scope Boundaries #

In scope

  • Six new greptile.json rules in printing-press-library (R1–R6).
  • One Python scan script under .github/scripts/verify-supply-chain/ in printing-press-library (R7).
  • One new CI workflow verify-supply-chain.yml in printing-press-library (R8).
  • AGENTS.md section + solutions doc in printing-press-library.
  • Mirror of the rules, script, and workflow into cli-printing-press (R9), with scope adaptations for the generator-repo surface.
  • Branch-protection request: add verify-supply-chain as a required check once a one-week green window has elapsed on each repo's main.

Not in scope

  • Runtime sandboxing of go install, npx, or any user-machine execution.
  • Sigstore/Fulcio signing changes for the npm wrapper (already uses OIDC Trusted Publishing — sufficient).
  • Mandatory 2FA enforcement on the org or per-account.
  • Slack/Discord alerting on scan findings.
  • Slack-based incident-response runbook.
  • Real-time dependency-age checks against the npm registry API (high false-positive rate; rate-limit risk).
  • Retroactive sweep of already-published CLIs for any pattern the new scan flags.
  • Replacing or disabling existing Greptile rules, govulncheck.yml, or any verify-* workflow.

Deferred to follow-up work

  • Parity-check script diffing signals.py between the two repos. Cheap to add, but with a single admin owning both repos and only two surfaces to keep in sync, deliberate coordination is sufficient for the first iteration. Revisit if a second contributor lands meaningful changes to either copy.
  • SHA256 IOC matching against known malicious payload hashes (e.g., the TanStack router_init.js hash). Useful once a hash-feed source is selected; out of scope here to avoid coupling the gate to an external feed.
  • Base64-array heuristics for obfuscator.io fingerprints. Real signal but high false-positive rate against legitimate bundlers; defer until we see a near-miss attack shape that would have caught it.
  • Egress blocking of known C2 hosts (e.g., .getsession.org) at the org perimeter. Outside this repo's surface; route to org-level network policy.

Key Technical Decisions #

KD1: Extend existing greptile.json rather than create a new config file

The repo already uses greptile.json with six customContext.rules entries. Append the new rules; do not introduce a parallel config. One config file, one review surface.

KD2: Two-layer defense, by signal type

Mechanical signals (regex matches, JSON key presence, specific string literals) go to the Python scan and hard-fail with structured GitHub Actions error annotations. Judgment signals (does this Go source actually read credentials in context? is this replace directive's stated purpose plausible?) go to Greptile rules that surface rationale in a PR comment. Each layer plays to its strength.

KD3: Path-scoped rules over global rules

Every new rule names a path glob. Avoids noise on generated output, minified files, and legitimate auth surfaces. Follows existing repo precedent (scope: ["library/**/.printing-press.json"] on every customContext rule today).

KD4: Treat the npm wrapper as a first-class blast-radius surface

A compromised @mvanhorn/printing-press reaches every user via npx. A compromised single Go CLI only reaches users who installed that slug. The npm-wrapper lifecycle-script rule (R5) is a hard-fail; per-CLI rules are tiered.

KD5: replace directive tiering (the ordertogo precedent)

The existing library/food-and-dining/ordertogo/go.mod has three replace directives pointing at ./third_party/... for legitimate vendoring. A blanket-block rule would either grandfather ordertogo via allowlist (drift risk) or fire on its next reprint (false alarm). Instead, tier by target shape:

  • block replace X => github.com/... or any URL/host target.
  • advise replace X => ./local/path or ../path. Greptile flags for review; scan emits notice, does not block.

This is consistent with the BufferZoneCorp attack shape (remote redirect to attacker fork) without false-positiving on legitimate local vendoring.

KD6: Coordinated landing, but staged main merges

The two repo PRs are authored together by the same person with admin access on both. Land the printing-press-library PR first, observe one week of clean runs, then land the cli-printing-press mirror. Rationale: false-positive calibration on the larger repo (more PR throughput, more diverse touch surface) before duplicating any miscalibration into the upstream generator. Both PRs are written in parallel; merge order is the discipline.

KD7: Required-check promotion happens after a green window, not at merge

The new workflow runs informationally on every PR for one week. After a week of clean runs (no spurious failures on legitimate PRs), promote to a required check via branch protection. Avoids day-one merge-blocker pain if a rule mis-fires on a real PR.

U1. Add mechanical-signal Greptile rules to printing-press-library

Goal: Extend greptile.json with six new customContext.rules entries covering R1–R6 attack shapes. Greptile reviews every PR and posts a comment with rationale; this is the judgment layer that explains why the rule fired even when the mechanical scan also blocks.
Requirements: R1, R2, R3, R4, R5, R6.
Dependencies: none.
Files:
  • greptile.json [mod]
Approach: Append six entries to customContext.rules. Each follows the existing rule shape: {"scope": [...path globs...], "rule": "..."}. The rule prose names the attack shape, the file shape that would trip it, and an explicit allowlist where one applies (e.g., npm-publish.yml for id-token: write). Greptile severity is implicit — rules are written as "Flag X" / "P0. Block X" matching the existing patterns.
  • R1 rule: scope .github/workflows/**. Detect any workflow that combines pull_request_target trigger with a checkout step whose ref references github.event.pull_request.head.sha, github.event.pull_request.head.ref, or refs/pull/<n>/merge. This is the TanStack root cause.
  • R2 rule: scope .github/workflows/**. Detect id-token: write anywhere except inside npm-publish.yml. Reference the OIDC Trusted Publishing allowlist.
  • R3 rule: scope library/**/go.mod. Detect new replace directives. Two tiers: hard-flag (P0) when the target is a URL/host (github.com/..., http*://...); advisory (P1) when the target is a local relative path. Reference the BufferZoneCorp attack shape.
  • R4 rule: scope .github/workflows/**. Detect GOPROXY, GOFLAGS, or GONOSUMCHECK appearing in env: blocks at workflow, job, or step level.
  • R5 rule: scope npm/package.json. Detect newly-added preinstall, postinstall, or prepare entries under the scripts object. Reference Axios and TanStack as primary motivators.
  • R6 rule: scope library/**/go.mod. When the file existed on origin/main, the module directive must continue to begin with github.com/mvanhorn/printing-press-library/library/. Flag any rewrite to an external module path.
Patterns to follow: The existing six entries in customContext.rules (library/**/.printing-press.json rules, the cli-skills/pp-*/SKILL.md P0 rule) — same prose voice (declarative, names the attack, names the allowlist).
Test scenarios:
  • Fixture PR adds a workflow with on: pull_request_target + steps: - uses: actions/checkout@v4 with: { ref: ${{ github.event.pull_request.head.sha }} } → Greptile flags R1 with TanStack reference.
  • Fixture PR adds permissions: { id-token: write } to a non-publish workflow → Greptile flags R2.
  • Fixture PR adds replace github.com/example/foo => github.com/attacker/fork v1.0.0 to library/payments/kalshi/go.mod → Greptile flags R3 hard-tier.
  • Fixture PR adds replace github.com/example/foo => ./vendor/foo to a library go.mod → Greptile flags R3 advisory-tier (does not demand block).
  • Fixture PR sets env: { GOPROXY: "https://mirror.example/" } in a workflow job → Greptile flags R4.
  • Fixture PR adds "postinstall": "node setup.js" to npm/package.json → Greptile flags R5.
  • Fixture PR rewrites the module line in an existing library/<cat>/<slug>/go.mod from canonical to github.com/attacker/fork/... → Greptile flags R6.
  • Fixture PR bumps a Go dependency in library/<cat>/<slug>/go.mod normally (no replace, no module change) → no rule fires.
Verification: jq . greptile.json validates as JSON; manual sandbox PR for each fixture above produces the expected Greptile comment within the existing 17-minute window the policy gate already waits on.

U2. Build Python scan script under .github/scripts/verify-supply-chain/

Goal: Implement the mechanical layer — a deterministic Python script that consumes the PR diff and emits structured GitHub Actions annotations for each signal, exiting non-zero on any block-severity finding.
Requirements: R1, R2, R3 (block tier only), R4, R5, R6, R7 (advisory severity), R8.
Dependencies: none (script can be developed and tested standalone before wiring into CI).
Files:
  • .github/scripts/verify-supply-chain/__init__.py
  • .github/scripts/verify-supply-chain/scan.py
  • .github/scripts/verify-supply-chain/signals.py
  • .github/scripts/verify-supply-chain/test_scan.py
Execution note: Develop test-first. Write test_scan.py against the fixture-PR shapes enumerated below before scan.py grows past a skeleton. This is a security gate — false negatives are silent and expensive.
Approach:
  • scan.py is the entry point: parses CLI args (--base-ref, --head-ref, optional --strict), computes the diff via subprocess.run(['git', 'diff', ...]), iterates files, dispatches each touched file to its signal matchers in signals.py.
  • signals.py defines the pattern catalog. Each signal is a small function or dataclass with: a path-glob predicate, a content predicate (regex or AST-light parsing for go.mod and YAML), a severity (block or advise), a one-line message template, and a one-line remediation hint.
  • Output format: GitHub Actions annotations — ::error file=...,line=...::<message> for block, ::notice file=...,line=...::<message> for advise. This is the same pattern verify_publish_package.py uses.
  • Exit code: 0 if no block-severity findings, 1 if any block-severity finding. Advisory findings do not affect exit code.
  • --strict flag escalates advisories to block; off by default. Used in dispatch-only debugging, not the workflow.
  • Signal implementations:
    • R1: YAML-parse touched .github/workflows/*.yml files; for each top-level on: with pull_request_target, walk jobs/steps looking for uses: actions/checkout with a with.ref value containing head.sha, head.ref, or refs/pull/.
    • R2: YAML-parse touched .github/workflows/*.yml files; emit error if permissions.id-token: write appears at workflow or job level and the filename is not npm-publish.yml.
    • R3: Line-diff parse touched library/**/go.mod; for each added line matching ^replace\s+\S+\s+=>\s+(\S+), classify target: starts with ./ or ../ → advise; contains :// or matches ^[a-z]+\.[a-z]+/ → block.
    • R4: Line-diff parse touched .github/workflows/*.yml; emit error if added lines define GOPROXY, GOFLAGS, or GONOSUMCHECK inside an env: block.
    • R5: JSON-parse npm/package.json at base and head; emit error for any key in scripts at head that is in {preinstall, postinstall, prepare} and was not in base.
    • R6: For each touched library/**/go.mod, fetch base copy via git show <base-ref>:<path>. If the file existed at base, extract module directive from both; emit error if head's value does not start with github.com/mvanhorn/printing-press-library/library/.
Patterns to follow:
  • .github/scripts/verify-publish-package/verify_publish_package.py — argparse, subprocess for git, pathlib.Path, dataclass Problem, GH-annotation escape helper, REPO_ROOT = Path(__file__).resolve().parents[3].
  • .github/scripts/verify-skill/verify_skill.py — predicate-style signal matchers, COMMON_FLAGS allowlist set pattern (mirror this for the npm-publish.yml allowlist).
Test scenarios:
  • R1 happy path: diff adds a workflow with safe pull_request_target and no checkout ref override → no findings. Covers AE1.
  • R1 attack shape: diff adds a workflow with pull_request_target + ref: ${{ github.event.pull_request.head.sha }} → block, exit 1.
  • R2 attack shape: diff adds id-token: write in verify-skills.yml → block, exit 1.
  • R2 allowlist: diff modifies npm-publish.yml with id-token: write already present → no finding.
  • R3 block tier: diff adds replace example.com/foo => github.com/attacker/foo v0.0.1 to a library go.mod → block, exit 1.
  • R3 advise tier: diff adds replace example.com/foo => ./vendor/foo to a library go.mod → notice, exit 0.
  • R3 existing-untouched: a PR that does not modify library/food-and-dining/ordertogo/go.mod produces zero findings even though that file contains three legitimate replace directives. Regression guard against scanning beyond the diff.
  • R4 attack shape: diff adds env: { GOPROXY: "https://attacker.example/" } to a workflow → block, exit 1.
  • R5 attack shape: diff adds "postinstall": "node ./malicious.js" to npm/package.json → block, exit 1.
  • R5 legitimate: diff modifies an existing legitimate prepublishOnly script (already present in npm/package.json) → no finding (not in the watched set).
  • R6 attack shape: diff rewrites module github.com/mvanhorn/... to module github.com/attacker/fork in an existing go.mod → block, exit 1.
  • R6 new CLI: diff adds a brand-new library/<cat>/<slug>/go.mod with canonical module path → no finding.
  • R6 new CLI bad path: diff adds a brand-new go.mod with non-canonical module path → block, exit 1.
  • Mixed clean PR: diff touches only Go source under an existing CLI's internal/cli/ with no go.mod changes, no workflow changes, no npm changes → no findings.
  • Strict mode: with --strict, R3 advise-tier promotes to block → exit 1.
Verification: python3 -m pytest .github/scripts/verify-supply-chain/test_scan.py -v passes all fixtures. python3 -m .github.scripts.verify-supply-chain.scan --base-ref origin/main --head-ref HEAD runs cleanly on a clean checkout of main against itself (zero findings).

U4. Add Greptile-only judgment rule for Go credential-path reads

Goal: Catch the node-ipc / TanStack credential-harvesting shape in Go source — a non-auth CLI that suddenly starts reading ~/.aws/credentials or harvesting tokens. This rule lives in Greptile only, never in the mechanical scan, because legitimate auth CLIs (GitHub API wrappers, AWS clients) read these paths intentionally.
Requirements: R7.
Dependencies: none.
Files:
  • greptile.json [mod]
Approach: One new customContext.rules entry. Scope: library/**/internal/**/*.go, library/**/cmd/**/*.go. Rule prose names the credential-path literals (.aws/credentials, .ssh/id_, .kube/config, .claude/, .vscode/ hooks) and the sensitive env vars (AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN, ACTIONS_ID_TOKEN_REQUEST_URL, GITHUB_TOKEN). The rule instructs Greptile to judge whether the CLI's purpose plausibly explains the credential read: a CLI named aws-foo-pp-cli reading .aws/credentials is fine; a CLI named weather-api-pp-cli reading the same path is a P0 finding.
Patterns to follow: The existing manuscript-content rule in greptile.json under "other" is the closest analog — judgment-heavy, prose-driven, scope-specific.
Test scenarios:
  • Sandbox PR adds os.Getenv("AWS_SECRET_ACCESS_KEY") to library/payments/kalshi/internal/client/client.go (Kalshi is not an AWS CLI) → Greptile flags P0.
  • Sandbox PR adds os.Getenv("AWS_SECRET_ACCESS_KEY") to a hypothetical AWS-shaped CLI → Greptile does not flag (judgment).
  • Sandbox PR adds os.Getenv("GITHUB_TOKEN") to a GitHub-shaped CLI → Greptile does not flag.
  • Sandbox PR adds a string literal "~/.claude/settings.json" in any CLI → Greptile flags (the TanStack persistence path; no CLI in this catalog should write there).
Verification: JSON validates. Manual sandbox PRs of each shape produce expected Greptile output. Existing AWS- or GitHub-shaped CLIs in the catalog do not produce spurious findings on unrelated PRs.

Risks & Dependencies #

RiskLikelihoodMitigation
False positives on legitimate replace directives. Medium (one CLI today has them). KD5 tiering: local-path targets are advise-only. Existing ordertogo directives don't trip the gate because the scan is diff-aware (only new lines count). Reprints regenerate the same lines verbatim.
False positives on legitimate auth surfaces (Go CLIs that genuinely read ~/.aws/credentials). Medium. U4 is judgment-only (Greptile), never mechanical-block. Greptile reads the CLI's purpose from SKILL.md / README.md / its name and grants context-appropriate findings.
Day-one merge blocking on legitimate PRs because of mis-tuned regex. Medium. KD7: new check runs informationally for one week. Promoted to required after a clean window. No branch-protection wiring on landing.
Drift between the two repos' signals.py over time. Low (single admin owns both). Vendor-source header in cli-printing-press/.github/scripts/verify-supply-chain/scan.py names date and source. Drift surfaces on next review of either repo's PR. Parity script deferred (see Scope/Deferred).
YAML parsing brittle on unusual workflow shapes. Low. Use pyyaml safe-load (already available in the verify-* scripts ecosystem). Test fixtures include the existing repo workflows as known-good inputs.
Greptile misinterprets rule prose and over- or under-flags. Medium. Two-layer design (KD2): mechanical scan is the authoritative block gate. Greptile is for rationale. A Greptile false-negative still loses the alert but does not let a block-tier attack through — the scan catches those.
Attacker pre-pollutes a repo and the gate misses it because the attack landed before the gate did. Low (audit confirmed clean baseline). Audit findings documented above (origin/main as of 2026-05-17 is clean for both repos).
The scan workflow itself becomes an attack surface. Low. Runs as pull_request (not pull_request_target), contents: read only, no id-token. Cannot be turned into a credential-exfil path on its own merits. Treat any PR modifying .github/scripts/verify-supply-chain/** as a Greptile P0 review surface (folded into U1's general workflow-trust prose).
